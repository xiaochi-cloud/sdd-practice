# 规范审查清单

> 用于审查 OpenSpec 规范质量

**使用时机：** 规范编写完成后，生成代码前  
**审查人：** 团队成员（非作者）  
**预计时间：** 10-15 分钟

---

## ✅ 完整性检查

### 元数据
- [ ] 名称清晰（能一眼看出功能）
- [ ] 版本号正确
- [ ] 描述完整（1-2 句话说明用途）
- [ ] 作者和日期填写

### 输入输出
- [ ] 所有输入都有定义
- [ ] 所有输出都有定义
- [ ] 类型明确（string/number/boolean 等）
- [ ] 必填/选填标注清楚
- [ ] 约束条件完整（长度、格式、范围等）

### 业务规则
- [ ] 业务规则无歧义
- [ ] 规则之间有优先级（如有冲突）
- [ ] 规则可验证/可测试

### 验收标准
- [ ] 验收标准明确
- [ ] 每个标准都可测试
- [ ] 覆盖正常场景和异常场景

---

## ✅ 一致性检查

### 命名
- [ ] 命名风格一致（如都用英文或都用中文）
- [ ] 术语使用一致（如"用户"vs"客户"）
- [ ] 字段名在全文档中一致

### 格式
- [ ] YAML 格式正确
- [ ] 缩进一致（2 空格或 4 空格）
- [ ] 注释清晰

### 引用
- [ ] 引用其他规范时路径正确
- [ ] 引用模型时名称一致

---

## ✅ 可验证性检查

### 约束可验证
- [ ] 所有约束都可自动化验证
- [ ] 没有模糊约束（如"合理的值"）

### 规则可测试
- [ ] 业务规则可写成测试用例
- [ ] 有明确的输入输出预期

### 验收可执行
- [ ] 验收标准可执行
- [ ] 有明确的通过/失败标准

---

## ✅ 可维护性检查

### 模块化
- [ ] 规范拆分合理（不过大或过小）
- [ ] 可复用部分已提取

### 注释
- [ ] 复杂逻辑有注释
- [ ] 业务规则有说明
- [ ] 示例充分

### 版本管理
- [ ] 版本号遵循语义化版本
- [ ] 变更有记录

---

## ✅ Java 开发者友好检查

### 类比清晰
- [ ] 用 Java 开发者熟悉的术语
- [ ] 复杂概念有类比

### 示例充分
- [ ] 有代码示例
- [ ] 示例可运行
- [ ] 示例覆盖主要场景

---
